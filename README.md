Практическая работа 19

В данной работе был создан REST API-сервер на Node.js с использованием Express. Для хранения данных использовалась реляционная база данных PostgreSQL. Была создана таблица users с полями: идентификатор, имя, фамилия, возраст, время создания и время обновления. Реализованы пять основных маршрутов: создание пользователя, получение списка всех пользователей, получение одного пользователя по идентификатору, обновление данных пользователя и удаление пользователя. Сервер запущен на порту 3000.



Практическая работа 20

В данной работе был создан аналогичный REST API-сервер, но вместо реляционной базы данных использовалась нереляционная база данных MongoDB. Для удобной работы с базой данных применялась библиотека Mongoose, которая позволяет описывать схемы данных. Была создана схема пользователя с теми же полями. Реализованы те же пять маршрутов для работы с пользователями. Сервер запущен на порту 3001.



Практическая работа 21

Данная работа была посвящена кэшированию. В существующее API было добавлено кэширование с помощью Redis — быстрого хранилища данных в оперативной памяти. Была написана универсальная функция, которая сначала проверяет наличие запрашиваемых данных в кэше. Если данные есть, они мгновенно возвращаются из Redis. Если данных нет, сервер обращается к основной базе данных, получает ответ, сохраняет его в Redis на определённое время и затем возвращает клиенту. Для списка пользователей кэш хранится одну минуту. Также при изменении данных кэш очищается. Сервер запущен на порту 3002.



Практическая работа 22

В данной работе была настроена балансировка нагрузки. Был создан простой бэкенд-сервер, который отвечает на запросы сообщением со своим портом. Запущены три экземпляра этого сервера на разных портах: 3003, 3004 и 3005. Затем был установлен и настроен Nginx. В конфигурационном файле была создана группа серверов, в которую добавлены все три бэкенда, причём третий сервер помечен как резервный. Nginx был настроен на прослушивание порта 80 и перенаправление всех входящих запросов на группу серверов с балансировкой по очереди. Резервный сервер активируется только когда оба основных сервера перестают отвечать.
